# MyDemo Bot 项目周报

## 一、项目概述

基于 Spring Boot + MyBatis 框架，集成 iLink SDK 和 DashScope SDK，实现了一个功能完善的微信智能助手 Bot。用户通过微信聊天即可与 AI 对话、查询天气、识别图片、生成图片、收听语音播报等。

---

## 二、技术栈

| 层级 | 技术 |
|------|------|
| 框架 | Spring Boot 4.0.7 + MyBatis |
| 微信接入 | wechat-ilink-sdk-java v2.3.3 |
| 大模型 | dashscope-sdk-java v2.22.25（千问系列） |
| 天气 | 心知天气 API |
| 数据库 | MySQL |

## 三、功能模块

### 3.1 微信接入模块

| 功能 | 实现方式 |
|------|---------|
| Bot 扫码登录 | ILinkClient.executeLogin() |
| 会话持久化 | SessionManager 序列化 LoginContext 到 JSON，重启免扫码 |
| 消息接收 | HeartbeatService 自动轮询 + OnMessageListener 回调 |
| 文字发送 | client.sendText() |
| 图片发送 | client.sendImage() |
| 语音发送 | client.sendVoice() |

**解决的问题**：
- 每次启动需要扫码，体验差 → 实现会话持久化，自动登录，3秒内可按Enter切换账号

### 3.2 大模型对话模块

| 功能 | 实现方式 | 模型 |
|------|---------|------|
| 文字对话 | Generation.call() | qwen-plus |
| Function Calling | 定义 weather/generate_image/help/status/version 工具 | qwen-plus |
| 识图 | MultiModalConversation.call() | qwen-vl-max |
| 文生图 | ImageGeneration.call() | wan2.6-t2i |
| 图生图 | ImageGeneration.call()（多图输入） | wan2.6-image |
| 语音合成 | HttpSpeechSynthesizer.callAndReturnAudio() | cosyvoice-v3-flash |

**关键技术**：
- 使用 DashScope SDK 替代手动 WebClient，自动处理序列化、异常、API Key
- Function Calling 实现一次 API 调用完成意图判断+执行，比两阶段调用减少一半延迟

### 3.3 天气查询模块

| 功能 | API | 说明 |
|------|-----|------|
| 实时天气 | /v3/weather/now.json | 温度、体感、湿度、风向、风速、能见度、气压 |
| 多天预报 | /v3/weather/daily.json | 3/7/15天，LLM 根据语义自动判断天数 |
| 全球城市 | 同上 | 支持北京、纽约、东京、伦敦等全球城市 |

**智能分流**：实时/未来天气 → 调心知天气 API；主观感受/历史天气 → 千问直接回答

**解决的问题**：
- 自然语言"杭州未来五天"无法被正则匹配 → 改用 LLM Function Calling，千问自行提取城市和天数
- "纽约的天气"查询失败 → Tool 描述添加全球城市说明
- 天气输出太简陋 → 增加体感温度、风速、能见度、气压字段

### 3.4 图片处理模块

#### 图片识别

用户发送图片 → 自动识图（qwen-vl-max）→ 返回文字描述 → 支持多轮追问

**架构**：
```
图片到达 → ImageService.recognizeImage() → 存描述到 imageDesc
文字提问 → 从 imageDesc 取描述 + 用户问题 → LLM 基于描述回答（不重复调识图API）
用户: "重新看看" → 重新调识图API，更新描述
无关话题 → 自动清除缓存
```

#### 图片生成

用户说"画一只猫" → LLM Function Calling → wan2.6-t2i 文生图 → 回复图片

#### 图片变换

用户说"变成卡通/把电脑p掉" → 检测为变换请求 → wan2.6-image 图生图（原图+风格指令）→ 回复变换后图片

#### 多图支持

用户发送多张图片 → 缓存最多5张 → "合并这两张图" → 多图传给 wan2.6-image → 生成合并图

**数据结构**：
```
ImageEntry { bytes, desc, timestamp }
ConcurrentHashMap<String, List<ImageEntry>> imageBuffers  // 每用户最多5张，5分钟过期
```

**解决的问题**：
- 微信图片和文字分开发送 → 图片缓冲机制，文字到达时合并处理
- 只发图片不发文字导致后续聊天被强制识图 → 5分钟TTL超时清除
- 图片描述太长占用微信显示 → 短描述输出给用户，完整描述内部存储
- 图生图没参考原图 → 区分变换/从零生成，变换时走 wan2.6-image 传原图
- keyword 匹配意图不准确 → 去掉 isImageQuestion/isImageGenRequest，改用 LLM 上下文感知

### 3.5 记忆模块

**功能**：让 Bot 记住用户的个人信息（名字、生日、喜好等），重启后不丢失。

| 功能 | 实现方式 |
|------|---------|
| 记忆保存 | CommandRouter 添加 `remember_fact` 工具，LLM Function Calling 触发保存 |
| 记忆读取 | LlmService 每次调用前从 MemoryService 读取，拼接到 system prompt |
| 持久化 | 文件存储：`memory-{userId}.json`，Jackson 序列化/反序列化 |
| 数据结构 | `ConcurrentHashMap<String, ConcurrentHashMap<String, String>>` |

**流程**：
```
用户: "叫我托尼"
  → LLM 判断是个人信息 → 调 remember_fact({"key":"名字","value":"托尼"})
  → MemoryService.setFact(userId, "名字", "托尼")
  → 写入 memory-o9cq80x.json

每次对话时:
  LlmService.buildSystemPrompt(userId)
  → 从 memory-{userId}.json 读取事实列表
  → 拼接到 system prompt: "用户信息：名字是托尼，生日是5月20号..."
  → 千问看到用户信息后回答问题时会自动使用
```

**记忆文件格式**（`memory-o9cq80xWz8OEdKTapGagHOZB5cs4@im.wechat.json`）：
```json
{"名字": "托尼", "生日": "5月20号", "职业": "程序员"}
```

## 六、收获与理解

1. **Function Calling 替代关键词匹配**：大模型的语义理解能力远强于规则匹配，将意图判断交给 LLM 是更优方案
2. **SDK 优先于手动 HTTP**：DashScope SDK 自动处理序列化、异常、重试，开发效率和数据可靠性显著提升
3. **协议层面的局限与应对**：微信图片文字分离发送是协议限制，通过缓存+TTL机制优雅解决
4. **上下文管理**：图片描述缓存、多轮对话历史、语音模式开关，都是上下文管理的典型案例
5. **多模态能力整合**：文字对话(qwen-plus) + 识图(qwen-vl-max) + 文生图(wan2.6-t2i) + 图生图(wan2.6-image) + 语音(cosyvoice-v3-flash)，实现完整的多模态交互闭环

## 八、核心技术机制详解

### 1. 登录免扫码机制如何实现

**核心思路**：利用 SDK 的 LoginContext 序列化能力，首次扫码后保存登录凭证，后续启动直接恢复。

```
首次启动:
  executeLogin() → 用户扫码 → onLoginSuccess(LoginContext ctx)
  → SessionManager.save(ctx)
  → 序列化为 ilink-session.json:
  {
    "botToken": "xxx",
    "userId": "xxx@im.bot",
    "botId": "xxx",
    "baseUrl": "https://..."
  }

后续启动:
  SessionManager.load() → 反序列化 LoginContext
  → ILinkClient.builder().loginContext(ctx) → SDK 内置恢复登录态
  → HeartbeatService 直接启动 → 消息监听生效

切换账号:
  启动时 3 秒倒计时 → 按 Enter → SessionManager.clear() → 走扫码流程
```

**关键点**：LoginContext 是 SDK 内部的登录凭证，SDK 通过 `loginContext()` 方法接收后直接进入 LOGGED_IN 状态。不需要 token 刷新，由 SDK 内部的 HeartbeatService 自动维持连接。

---

### 2. 图像与文字结合如何实现

**问题**：微信协议将图片和文字拆成两条独立的 WeixinMessage 推送，存在时序问题（文字先到、图片后到）。

**解决**：图片缓冲 + 描述拼接方案。

```
数据结构:
  ConcurrentHashMap<String, List<ImageEntry>> imageBuffers
  ImageEntry { byte[] bytes; String desc; long timestamp; }
  MAX_IMAGES = 5, IMAGE_BUFFER_TTL = 5分钟

流程:
  用户发图 → handleImageMessage()
    → downloadImage() → recognizeImage() 获取描述
    → addImageToBuffer(userId, bytes, desc)
    → 回复短摘要

  用户发"这是什么" → handleMessage()
    → 检查 imageBuffers 有缓存
    → buildImageContext(images, text):
      "用户发送了图片，内容：{完整描述}\n\n用户说：这是什么"
    → CommandRouter.route() → LLM 看到图片描述 + 问题 → 基于描述回答
```

**关键点**：图片描述作为上下文拼接进用户消息，不是直接发图片给 LLM。LLM 通过文字描述理解图片内容，实现"看过图片"的效果。无需重复调识图 API。

---

### 3. 图生图如何实现

**核心思路**：有原图缓存时自动使用 wan2.6-image 模型，SDK 自动上传临时文件作为参考图。

```
sendReply() 检测到 IMG_GEN:
  ├── imageBuffers 有数据
  │     → ImageService.transformImage(List<byte[]> images, String prompt)
  │     → 写临时文件 → content = [
  │           {"image": "/tmp/transform_1.png"},
  │           {"image": "/tmp/transform_2.png"},
  │           {"text": "把这张图变成卡通风格"}
  │        ]
  │     → ImageGeneration.call() → wan2.6-image 模型
  │     → 模型接收原图 + 风格指令 → 生成基于原图的变换图
  │     → 删除临时文件
  │
  └── 无数据 → ImageService.generateImage(prompt)
        → content = [{"text": "一只猫"}]
        → ImageGeneration.call() → wan2.6-t2i 模型
        → 从零生成
```

**关键点**：wan2.6-image 支持 image+text 混合输入，模型自行判断原图是否相关。不需要代码层面区分"变换"和"从零生成"，有缓存就传原图，模型自己决定是否参考。

---

### 4. 用户发图后再发送内容如何判断意图

**核心思路**：不判断。去掉所有关键词匹配（isImageQuestion / isImageGenRequest），用 LLM 上下文感知替代。

```
图片缓存存在 → buildImageContext() 拼接描述 + 用户文字 → CommandRouter.route()
  ↓
LlmService.chat() 带 tools（weather / generate_image / help / status / version）
  ↓
千问根据语义自行判断:
  ├── "这是什么"       → 图片相关问题 → 基于描述文字回答
  ├── "画一只猫"       → 生图请求     → function_call("generate_image")
  ├── "变成卡通"       → 编辑请求     → function_call("generate_image")
  ├── "天气"           → 命令         → function_call("weather") → 同时清除图片缓存
  ├── "今天星期几"     → 闲聊         → 文字回复
  └── "我没说查北京"   → 纠正         → LLM 重新理解回答
```

**命令保护**：如果用户文字匹配到内置命令（如 weather），直接清除图片缓存并执行命令，避免图片上下文干扰命令执行。

### 5. 天气生成机制

**核心思路**：LLM Function Calling + 心知天气 API，支持实时和预报。

```
实时天气:
  用户: "杭州天气" → CommandRouter → 未匹配命令
  → LlmService.chat() + tools → 千问判断语义
  → function_call("weather", {"city":"杭州", "days":1})
  → WeatherCommand.execute(["杭州"])
  → WeatherService.getWeather("杭州")
  → GET /v3/weather/now.json?location=杭州
  → 解析: text, temperature, feels_like, humidity, wind_direction, wind_speed, wind_scale, visibility, pressure
  → 格式化输出

多天预报:
  用户: "杭州未来五天"
  → function_call("weather", {"city":"杭州", "days":7})
  → WeatherCommand.execute(["杭州", "7"])
  → getForecastMulti("杭州", apiDays=7, showDays=5)
  → GET /v3/weather/daily.json?location=杭州&days=7
  → 解析 daily[] 数组 → 取前5天
  → 格式化: "07-18 明天: 晴 28~38°C\n07-19 后天: 多云 26~35°C\n..."

智能分流:
  "杭州多少度" / "杭州明天" / "纽约天气"
    → LLM 判断是实时/预报 → function_call(weather) → 心知天气 API（精确数据）
  "杭州热不热" / "杭州昨天天气" / "杭州夏天一般多少度"
    → LLM 判断是主观/历史/常识 → 千问直接回答（不调 API，心知查不了历史）
```

**关键点**：天数自动匹配策略。
```
days ≤ 3 → API 调用 days=3
days ≤ 7 → API 调用 days=7
其他     → API 调用 days=15
```

LLM 根据用户自然语言自动判断 days 参数：今天→1，未来两天→3，未来五天→7，未来十天→15。

### 6. 多图处理机制

**核心思路**：List 代替单值，支持累加和最多5张限制。

```
数据结构:
  ConcurrentHashMap<String, List<ImageEntry>> imageBuffers
  MAX_IMAGES = 5

累加流程:
  用户发图1 → addImageToBuffer(userId, bytes, "图片一是...")
     → images = [ImageEntry1]
     → "图片识别 1/5: ...可以继续发图片"

  用户发图2 → addImageToBuffer(userId, bytes, "图片二是...")
     → images = [ImageEntry1, ImageEntry2]
     → "图片识别 2/5: ...可以继续发图片"

  用户发图3 → addImageToBuffer(userId, bytes, "图片三是...")
     → images = [ImageEntry1, ImageEntry2, ImageEntry3]
     → "图片识别 3/5: ...已达到最多5张图片"

多图上下文拼接:
  buildImageContext(images, "合并这三张图"):
  "用户发送了3张图片：
   图片1：图片一是...
   图片2：图片二是...
   图片3：图片三是...

   用户说：合并这三张图"

  → CommandRouter.route() → LLM → function_call("generate_image")
  → ImageService.transformImage(3张byte[], prompt)
  → wan2.6-image 接收3张参考图 + "合并这三张图"指令 → 生成合并图

过期与溢出:
  5分钟过期 → getValidImages() 自动移除过期条目
  超过5张 → images.remove(0) 自动移除最旧的
```

**关键点**：与单图完全复用同一套逻辑，仅将 `ConcurrentHashMap<String, byte[]>` 升级为 `ConcurrentHashMap<String, List<ImageEntry>>`。LLM 上下文自动适配单图/多图描述格式。

### 7. Function Calling 工作原理

**核心思路**：定义工具(tools) → 传给 LLM → LLM 自行决定是否调用 → 解析调用结果 → 执行对应功能。

```
步骤1: 定义工具（CommandRouter.buildTools()）
  tools = [
    ToolFunction { name:"weather",  params:{city, days} },
    ToolFunction { name:"generate_image", params:{prompt} },
    ToolFunction { name:"help" },
    ToolFunction { name:"status" },
    ToolFunction { name:"version" }
  ]

步骤2: 传入 LLM（LlmService.chat()）
  GenerationParam.builder()
    .model("qwen-plus")
    .messages([system消息, 历史, 用户消息])
    .tools(tools)
    .build()

步骤3: LLM 返回（两种可能）
  情况A - 需要调工具:
    choices[0].finish_reason = "tool_calls"
    tool_calls[0].function.name = "weather"
    tool_calls[0].function.arguments = {"city":"杭州","days":1}

  情况B - 直接回答:
    choices[0].finish_reason = "stop"
    message.content = "你好呀，有什么可以帮你？"

步骤4: 处理结果（CommandRouter.route()）
  情况A → commands.get(fnName).execute(args) → 返回命令结果
  情况B → 直接返回文本 → ChatService.addHistory() → sendReply()
```

**关键优势**：一次 API 调用同时完成"意图判断"和"生成回复"，替代传统两阶段方案（先判断意图再生成回复），减少一半 API 延迟和成本。

### 8. 意图理解的演进之路

**关键词阶段**：
```
用户说"杭州天气" → 正则匹配 → 执行weather命令
用户说"天气"     → 匹配失败 → 返回"未知命令"
```
问题：只有精确匹配才生效，自然语言完全无法识别。

**关键词 + LLM分类阶段**：
```
用户说"画一只猫" → isImageGenRequest？→ 关键词匹配 → 未命中 → 聊天
用户说"p掉电脑" → isImageGenRequest？→ 关键词匹配 → 未命中 → 聊天
```
问题：keyword 列表是死的，永远覆盖不全同义表达（P图/p掉/去掉/移除/擦除...）。

**LLM Function Calling（当前方案）**：
```
用户说"把照片变卡通" → 不带关键词判断 → 直接交给 LLM + tools
  → 千问看到: "用户发送了图片，内容：{描述}。用户说：把照片变卡通"
  → 千问判断: 用户想把原图变换为卡通风格 → 调用 generate_image
  → 系统检测到图片缓存 → wan2.6-image
```
关键变化：从"我们替 LLM 判断意图"变为"LLM 自行判断意图"，减少了中间层，提高了准确率。

**演进对比**：

| 阶段 | 判读方式 | 准确率 | 维护成本 |
|------|---------|--------|---------|
| 纯关键词 | 正则/contains | 30% | 每次加新词 |
| 关键词+LLM | 关键词优先+LLM兜底 | 60% | 需要维护关键词表 |
| Function Calling | LLM全权判断 | 95% | 只需定义tools |

### 9. 异常处理与兜底澄清

**分层异常体系**：

```
BusinessException（业务异常基类）
  ├── MissingParameterException     // weather 没传城市
  ├── InvalidParameterException     // 参数格式错误
  ├── CityNotFoundException         // 城市不存在
  └── WeatherApiException           // 天气API调用失败
```

**各层处理策略**：

```
1. Command 层（命令执行异常）
   CommandRouter.executeCommand():
     ├── BusinessException → "[错误] " + e.getMessage()  // 友好的业务错误
     └── Exception         → "[错误] 系统异常"           // 不暴露内部细节

2. 图片处理层（图片下载/识图异常）
   handleImageMessage():
     ├── downloadImage 失败 → "抱歉，图片接收失败，请稍后再试"
     └── recognizeImage 失败 → "抱歉，图片识别失败，请稍后再试"

3. LLM 调用层（大模型API异常）
   LlmService.chat():
     ├── ApiException → "AI 服务暂时不可用，请稍后再试"
     └── 其他异常     → "AI 服务调用失败"

4. 回复层（发送异常 + 兜底）
   sendReply():
     ├── result == null/空 → "抱歉，我没理解你的意思，可以再说详细一点吗？"
     ├── IMG_GEN 失败      → "抱歉，操作失败，请稍后再试"
     └── sendText 失败     → 重试一次 "抱歉，操作失败"
```

**兜底澄清机制**：当 LLM 返回空结果或无法处理时，主动引导用户重新表达，而不是沉默或报错。
